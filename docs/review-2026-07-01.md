## Dimension assessments

### physics

The v2 physics core is methodologically sound: the cost bundle {aRoll, aAero,
beta, climbThr, abRatio, epsOffset} is derived once in app.js from physical
inputs (with flat speed solved from a wheel-power balance), coefficients are
genuinely kJ/m as the UI claims, edge costs are strictly positive in every mode
(the descent credit ε·β·|Δh| is mathematically bounded below the resistive cost,
so the downhill floor never yields free energy and Dijkstra optimality is
guaranteed), round-trip = forward + reverse independently-optimal legs is the
right answer to the stated question with consistent leg/total budget masking,
and the JS/Rust/graph engines are verifiably identical (v2Edge ≡ stepCost ≡
v2_edge). Geographic cell geometry (dx·111320·cos(latRef), dy·110574, diagonal =
hypot) is adequate at the flagship DEM's actual 0.7°×0.43° extent (~0.2% error).
The fragile parts are at the edges of the method and in the documentation: all
three user-facing descriptions of the model (README, in-app help theory section,
JSON-LD metadata) still document the retired v1 α/β/η joule-scale model that v44
replaced; the A* heuristic used by top-N routes is provably inadmissible under
the v2 descent credit (numerically confirmed: overestimates up to 3.3× on
1.1–6.2% descents) while its comment claims admissibility; and the kJ energy
budget is applied unconverted to inverted costs in maximize mode, silently
emptying results. Empirical constants (ε offset 0.13, k_smooth) cite bicycling-
energy-model/notas.md, which is not in this repo, so their provenance is not
reproducible from the codebase alone.

### engine

The engine core is in good shape: dijkstra(), densityField() and astar() all
carry BOTH load-bearing settled guards (stale-pop filter at energy-
worker.js:229/476/631 and never-relax-settled at 263/486/655), and the portal
relax sites in dijkstra (298) and densityField (501) repeat the same
guard/budget/maximize handling. The monotone radix heap is sound — pops come
only from bucket 0 (exact-equal keys), every push is g+edge with edge clamped >=
0 so no priority can fall below the bucket floor — and the targeted reset is
complete for real runs because with maxSettled=0 every relaxed cell eventually
settles into `order`. Portals are correctly excluded from A*, the layered DP,
and maximize mode (1365); buildPortalAdj's fwd/bwd pairing matches the reverse-
mode dh flip; the DP's memory cap precedes allocation and its backtrack
direction is correct. Message hygiene is solid: every handler branch posts
done/error, pre-try exceptions surface via w.onerror -> computeFailed, and the
density pool spreads baseMsg (portal arrays structured-cloned) while
transferring only per-worker fresh buffers from buildComputeGrid(); node test-
worker-pool.mjs passes on the current tree. The one genuine algorithmic defect
is the A* top-N heuristic: it is inadmissible under the v2 descent-recovery
model (empirically confirmed suboptimal routes), contradicting its own
admissibility comment; secondarily, top-N always scores seed->goal forward cost
regardless of the to/round field mode, so displayed route energies can disagree
with the field semantics.

### rust-parity

The Rust backend is a genuinely faithful port of energy-worker.js on the paths
the suite covers: v2_edge is an operation-order-identical copy of v2Edge (same
climb-threshold aero drop, same downhill eps clamp order, same final <0 clamp);
f32 energy storage with f64 heap priorities and the widened `tentative <
s.e[n_idx] as f64` compare, the dual settled guards, round-mode include-mask
semantics (f64 sum compared to the cap BEFORE `(s as f32)` rounding, matching
`Math.fround(fi+bi)`), portal building (same NaN→DEM-height fallback, same

the CLAUDE.md HashMap insertion-order worry holds because PortalAdj is only ever
accessed via keyed `get`, never iterated), Acc stays f64, the 37/55 bytes-per-

scratch + 20 acc; round 2·17+20+1), and /health's mem_budget_bytes + cores feed

n_slices formula. `cargo build && node test-backend.mjs` passes today with
max|Δenergy|=0 and max|Δpasses|=0 on every case. Coverage: /density is tested
for from/to/round × {no budget, eMax} × ±portals + round-total ±portals + one
plain maximize case; /single is tested for from/to/round × budget × ±passes ×
±network × ±portals + round-total+passes — so reverse portals ("to"),
round+portals, and the network eff_mask path on /single are all covered. NOT

max_edge_cost height-range being derived from the effective (network-ANDed) mask
in Rust but the raw DEM mask in JS — reachable from the UI and producing a
wholly different field — plus a documented-invariant gap where /single passes
accumulate in f32 instead of the JS Float64. Overall quality is high; the
fragility is concentrated in the untested maximize+network corner.

The graph engine is well-built where it counts: stepCost is an exact operation-
order mirror of energy-worker.js v2Edge, budget semantics (t > eMax, round

### Details

#### C0 [medium] A* heuristic for top-N routes is inadmissible on moderate descents; code comment claims admissibility

`energy-worker.js:563` — The heuristic h = aRoll·straight + beta·max(0, hGoal−h) assumes aRoll·dist is
the minimum per-distance edge cost, but the v2 descent credit eats into the
rolling term: for descent grades between ~1.1% and ~6.15% (defaults:
abRatio=0.0187, epsOffset=0.13, kSmooth=1) the true edge cost is BELOW
aRoll·dist. Verified numerically with the exact readCost() defaults: at 1.87%
grade a 100 m descent edge costs 0.184 kJ vs heuristic 0.607 kJ (3.3×
overestimate); inadmissible for all grades 1.5–6.15% tested. Failure scenario:
src on a ridge, dst below, with the cheapest route down a gentle 2–5% descent
corridor — A* inflates f for the corridor's nodes, settles dst via a costlier
flatter/steeper route first, and top-N 'route 1' (presented as optimal) reports
more energy than the exact Dijkstra destination path the same points produce
with top-N off. Only astar() is affected (from/to fields and destination paths
use exact Dijkstra); the same flaw existed in v1 (unsaturated v1 descent cost =
0.13·β·|Δh|, also < α·dist), so the comment's appeal to v1 is also wrong.

#### C1 [medium] Energy budget (kJ) is applied unconverted to inverted costs in maximize mode, silently emptying the field

`energy-worker.js:281` — dijkstra/astar/densityField prune with `tentative > eMax` where, under maximize,
`tentative` accumulates inverted costs (maxEdgeCost − cost) — and maxEdgeCost
includes beta·(global height range), e.g. ≈341 kJ per edge on a 450 m-relief DEM
(0.7585 kJ/m × 450). app.js passes the user's kJ budget unconditionally
alongside maximize (line 5721/5729: 'mode, cost, eMax, eMaxMode, … maximize').
Failure scenario: user ticks 'Maximizar energia' with a realistic budget of 300
kJ on sampa_geral → the very first inverted edge (~341 kJ) exceeds the budget →
every neighbour is pruned and the result is an empty field around the seed, with
no warning. Even when it doesn't empty the field, the budget degenerates into an
undocumented hop-count cap (~eMax/maxEdgeCost edges) instead of an energy limit.
Neither help.p.maximize nor help.p.budget_mode discloses that budget units
change meaning under maximize.

#### C2 [medium] In-app help modal still documents the retired v1 α/β/η model with joule-scale defaults that contradict the shipped v2 engine

`app.js:490` — The 'Modelo de custo assimétrico' theory section of the help modal (rendered at
index.html:1512-1515) shows the v1 formula 'subida: α·dist + β·Δh / descida:
max(0, α·dist − η·β·|Δh|)' and defaults 'α = 0.008 (8 mJ/m horizontal), β = 1 (1
J/m de subida), η = 0.1' — none of which exists since v44: the engine charges
aRoll + conditional aAero + β·Δh uphill, computes ε from grade (min(1,
abRatio/s) − 0.13) rather than a user η, has no η knob, and works in kJ (β ≈
0.759 kJ/m at defaults, not 1 J/m — a 1000× unit discrepancy plus a wrong
functional form). The adjacent help.p.params section correctly describes v2, so
the modal contradicts itself. Failure scenario: a user (or reviewer) validating
a computed field against the documented formula gets numbers off by orders of
magnitude and a wrong descent law; anyone reading 'η = 0.1 (10% of the climb is
recovered)' will misinterpret every descent cost in the field.

#### C3 [medium] README 'What it computes' and the JSON-LD featureList document the v1 α/β/η model and its defaults, not the shipped v2 model

selling point is following exact vector geometry. A metre-based tolerance
(converted via dxM/dyM) would decouple graph fidelity from DEM resolution.

## Method

45 agents: 2 test runners (all node suites; cargo build + JS↔Rust bit-parity),
9 parallel review dimensions (physics/methodology, worker engine, Rust parity,
app.js state machine, the v48 diff, data pipeline & hygiene, mirror sync,
PWA/deploy/ops, graph engine), then one adversarial verifier per finding (two
for highs) instructed to refute. Only findings that survived verification with
a concrete failure scenario are listed as confirmed.
